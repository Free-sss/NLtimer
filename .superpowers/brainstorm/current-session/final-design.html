📋 最终设计方案：UI 组件调试页面

汇总所有决策点，确认后进入实现

一、架构概览

// 依赖方向（单向）
 
core/designsystem ← 定义 DebugComponent / Registry 协议
↑
各 feature 模块 ← src/debug/ 下注册自己的组件
↑
feature/debug ← 新模块，收集注册组件，提供调试页面
↑
app (debugImplementation) ← debug 编译才包含，release 完全剥离

二、关键决策汇总

决策点 选择 理由
入口控制 Build Variant (C) debugImplementation 全自动，release 零残留
组件暴露方式 注册扩展函数 (B) 业务零耦合，debug 只依赖协议接口
注册触发时机 Application.onCreate 显式调用 (B) 组装点集中，时机可控，无魔法依赖
协议存放位置 core/designsystem 所有 feature 已依赖此模块
注册代码位置 各 feature 的 src/debug/java/ release 编译时根本不存在

三、文件变更清单

📁 core/designsystem — 新增 1 个文件
DebugComponent.kt DebugComponent 数据类 + DebugComponentRegistry 单例
📁 feature/debug — 新建模块，3 个文件
build.gradle.kts 模块构建配置，依赖 core/designsystem
DebugRoute.kt Composable 导航入口
DebugPage.kt 主页面：左侧分组 + 中间列表 + 右侧预览
📁 各 feature — 各新增 1 个注册文件（src/debug/java/）
feature/home HomeDebugComponents.kt — 注册 AddBehaviorSheet 等
feature/sub SubDebugComponents.kt
feature/stats StatsDebugComponents.kt
feature/... 同 pattern
📁 app — 修改 3 个文件
build.gradle.kts + debugImplementation(projects.feature.debug)
NLtimerApplication.kt onCreate 中显式调用各模块注册
NLtimerNavHost.kt + composable("debug") { DebugRoute() }

五、入口触发流程

App Drawer → 🐛 调试 菜单项
↓ (仅 debug 构建可见)
navController.navigate("debug")
↓
DebugRoute() → DebugPage()
↓
从 Registry 读取已注册组件 → 渲染三段式布局

✅ 设计保证

  • Release 构建：feature/debug 完全不存在，APK 零膨胀
  • Feature 模块：只在 src/debug/ 下多一个注册文件
  • 业务代码：无任何 if-debug 判断，无耦合
  • 扩展：新组件只需一行 register() 调用

⚠️ 约定

  • 各 feature 需要在 src/debug/java 下提供注册文件
  • Application.onCreate 需维护各模块注册调用
