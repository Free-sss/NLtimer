方案 C 详解：通过 Build Variant 控制入口

debug 版本自动可见调试页面，release 版本完全不存在

核心机制

// build.gradle.kts — 零配置，利用 Android 内置机制
// debug 变体自动拥有 debugImplementation 依赖
// release 变体编译时完全不包含 debug 模块代码
 
dependencies {
debugImplementation(projects.feature.debug)
}
 
// feature/debug 模块只在 debug 编译时存在
// release 时完全剥离，无任何残留

模块结构

📦 feature/debug
src/main/java/
└─ com/nltimer/feature/debug/
   ├─ DebugRoute.kt
   ├─ DebugComponentRegistry.kt
   └─ DebugComponentPage.kt
📱 app 模块集成
Drawer 菜单项
└─ if (BuildConfig.DEBUG)
      显示 "🐛 调试" 项
   
NavHost 路由
└─ debugImplementation
      composable("debug")

组件注册机制

// DebugComponentRegistry — 组件只需注册即可自动出现在列表中
object DebugComponentRegistry {
val components = listOf(
DebugComponent("AddBehaviorSheet", "新增行为弹窗") { AddBehaviorSheetPreview() },
DebugComponent("TagChip", "标签芯片") { TagChipPreview() },
DebugComponent("EmptyState", "空状态视图") { EmptyStatePreview() },
// 🔥 添加新组件只需加一行
)
}

✅ 优点

  • Release 版本零残留，代码完全不存在
  • 无需额外的开关逻辑，利用 Android 原生机制
  • 调试代码可自由引用任意模块的 UI 组件
  • 组件注册简单，一行代码加入调试列表
  • 与现有 Gradle 约定插件体系完美契合

⚠️ 注意事项

  • 切换 debug/release 需要重新构建
  • 调试模块不能引用各 feature 的内部实现（除非暴露 Preview 函数）
